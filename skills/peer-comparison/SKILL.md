# peer-comparison

The "relative context" skill. A ticker looks expensive or cheap only relative to something — this skill builds the "something" carefully, annotates direction-of-goodness for every metric, and flags business-mix differences that invalidate naive comparison.

## When to use

Loaded by the `/stockwiz` orchestrator at Stage 2c in deep-dive mode (single-ticker peer framing from SWS competitor snowflakes). The skill also documents a multi-ticker mode reserved for a future `/stockwiz-compare` command — see Roadmap in README. The multi-ticker spec is dormant; only single-ticker mode is currently exercised.

## Core principle

**Relative comparison requires business-mix awareness.** NVIDIA's P/E of 38 means something very different from AMD's P/E of 40 — similar surface numbers, wildly different underlying mixes. This skill's job is to produce comp tables AND caveat them honestly.

## Inputs to read

**For single-ticker mode (called from `/stockwiz`):**
- `raw/simply-wall-street.md` — SWS competitor snowflakes (free context on 2–5 peers including SWS scoring)
- `raw/finviz-snapshot.md` — sector and industry tags for the target ticker
- `raw/stockanalysis.md` — the target's metrics for the comp table rows
- `raw/sec-edgar-10k.md` — any peer mentions in business description (when 10-K prose is parsed in (roadmap))
- `analysis/fundamental.md` — the target's consolidated numbers (if written before this skill runs)

**For multi-ticker mode (dormant — reserved for a future `/stockwiz-compare` command):**
- Per-ticker raw files under each ticker's subdir
- Apply the same comp-table structure across all tickers
- Not currently exercised; spec preserved here as part of the roadmap

**You do NOT fetch peer data on your own.** The deep-researcher already pulled SWS's competitor snowflakes for free context. Per-peer deep-dives are a user-initiated action (running `/stockwiz <peer>` for each).

## Output structure

Write `SESSION_DIR/analysis/peer-comp.md` (single-ticker) or `SESSION_DIR/analysis/comp-table.md` (multi-ticker mode).

```markdown
# Peer Comparison — <TICKER>

*Generated by peer-comparison skill. Peers sourced from <source file(s)>.*

## Selected Peers

Rationale for each peer:
- **<PEER1>** (company name) — Why it's a peer: same sub-industry (Semiconductors), similar AI accelerator exposure, similar market cap tier. Differences that matter: more leveraged balance sheet, different customer concentration. Source: `raw/simply-wall-street.md` competitor snowflakes.
- **<PEER2>** — ...
- (2–5 peers typical; fewer if data is thin)

If only SWS competitor snowflakes are available, you may have 2–3 peers. If SWS did not provide peers (source failed), note it and present the target's metrics in absolute terms only, skipping the comp table.

## Comp Table

Every row must use the same columns. No cherry-picking. Missing cells show "—".

| Metric | <TARGET> | <PEER1> | <PEER2> | <PEER3> | Direction |
|---|---|---|---|---|---|
| Market cap | $4.58T | ... | ... | ... | n/a |
| Revenue TTM | $215.9B | ... | ... | ... | ↑ better |
| Revenue growth YoY | +65.5% | ... | ... | ... | ↑ better |
| Gross margin | 71.1% | ... | ... | ... | ↑ better |
| Operating margin | 60.4% | ... | ... | ... | ↑ better |
| Net margin | 55.6% | ... | ... | ... | ↑ better |
| ROIC | 126.3% | ... | ... | ... | ↑ better |
| ROE | 101.5% | ... | ... | ... | ↑ better |
| FCF margin | 44.8% | ... | ... | ... | ↑ better |
| Debt/Equity | 0.07 | ... | ... | ... | ↓ better |
| P/E (trailing) | 38.5 | ... | ... | ... | ↓ cheaper |
| P/E (forward) | 22.7 | ... | ... | ... | ↓ cheaper |
| EV/EBITDA | 34.0 | ... | ... | ... | ↓ cheaper |
| P/S | 21.2 | ... | ... | ... | ↓ cheaper |
| SWS Value score | 4 | ... | ... | ... | ↑ better |
| SWS Future score | 6 | ... | ... | ... | ↑ better |

Direction column indicates which way is favorable ONLY. Don't color the cells green/red in the markdown (the HTML report does that in muted shades). Never use "cheap" or "expensive" prose, use "trading below/above peer median on P/E".

## Relative Positioning

One paragraph describing where the target sits in the peer set. Example: "On growth metrics (revenue YoY, EPS YoY) NVDA is above all three comparables; on valuation multiples it is in the middle of the set; on SWS Value score it matches the peer median; on quality (ROIC, margins) it is above all peers." Just the shape, not advice.

## Caveats

**Mandatory.** Every peer comparison has caveats; never omit this section.

- Business-mix differences that matter: "AMD derives more revenue from CPUs and gaming than NVDA, so margin comparisons are not apples-to-apples."
- Time-period differences: "TSMC reports on a different fiscal calendar; metrics are aligned as of their latest filed quarter, which may lag NVDA by up to 3 months."
- Size differences: "ARM Holdings is 1/20th of NVDA's market cap; multiples at different scale have different meanings due to mix-shift leverage."
- Currency differences (if applicable for multi-market peers)
- Accounting differences (if applicable)

## Unknowns

- Private peers not covered (e.g. if target has material private competition like Anthropic, xAI, etc., flag it)
- Segment-level mix ((roadmap) with 10-K parsing)
- Customer overlap that would distort competitive framing
```

## Hard rules

1. **Same columns across rows.** No cherry-picking metrics per ticker. If a peer doesn't disclose a metric, the cell is "—".
2. **Every cell is dated.** In the markdown, you don't have to date every cell individually (the section header can note "as of {fetch date}"), but if metrics are from different fiscal periods for different peers, disclose it in Caveats.
3. **Direction-of-goodness annotated.** Every row in the comp table has a Direction column showing which way is favorable. ↑ better (higher is good), ↓ better (lower is good), or n/a (market cap, ticker name).
4. **No banned imperative language.** See `../report-generation/references/compliance-rules.md` § "Pre-filter guidance for skill authors". Peer-comparison-specific phrases to avoid: **"cheap"** / **"expensive"** without comparatives — "trading at a P/E below the peer median" is allowed; "cheap" alone is compliance-hazardous. Source labels like "Strong Buy" on a peer get `<q>` wrapping at report time; don't generate unquoted advisory prose yourself.
5. **Caveats section is mandatory.** If you can't think of any caveats, you haven't thought hard enough.
6. **Thin peer sets are honest.** If SWS competitor snowflakes returned 2 peers, you have 2 peers. Don't fabricate more. Note the thinness in Caveats.

## Known limitation

Peers come for free from SWS's competitor snowflake section (which is what stockwiz's deep-researcher already fetches). This typically gives 2–3 peers with SWS scoring but not deep metrics on each — you may have a comp table where the target has detailed stockanalysis.com data but the peers only have SWS snowflake scores.

That's OK for an initial version. A future phase will add a proper "peer fetch" capability where, when the user runs `/stockwiz <target>`, the system identifies 3–5 peers and issues lightweight Finviz fetches for each to build a full-metric comp table.

The thin-peer mode should produce a table like:

| Metric | NVDA | AMD | ARM |
|---|---|---|---|
| SWS Value | 4 | 2 | 1 |
| SWS Future | 6 | 6 | 6 |
| SWS Health | 6 | 5 | 6 |
| Revenue TTM | $215.9B | — | — |
| P/E (trailing) | 38.5 | — | — |
| ... | ... | — | — |

With "—" for missing cells and a caveat noting that only SWS-scored metrics have full peer comparability.
